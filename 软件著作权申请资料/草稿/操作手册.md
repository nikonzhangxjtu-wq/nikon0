## 二、说明

多模态智能客服Agent系统是一款面向电商及消费电子领域的智能客服软件。系统以产品说明书知识库为核心，结合检索增强生成、多模态图像理解和对话记忆等技术，为消费者提供产品使用咨询、故障排查指导和售后工单受理等客服服务。软件版本 V1.0。

系统适用的业务场景主要有三类。一是消费者查询家电、数码、运动器械等产品的使用说明和操作步骤，例如询问空气炸锅的清洁方法或洗衣机的程序设置。二是消费者上传产品故障照片后，系统识别图片内容并结合产品手册给出排查建议。三是当消费者发起报修、退货、退款等售后请求时，系统自动进入信息收集流程，通过多轮对话逐步获取工单所需信息。

本手册面向系统使用人员，按功能模块说明各项操作的方式、输入要求、操作步骤和预期结果。手册中涉及的命令均在终端中执行，使用者需要具备基本的命令行操作能力。

## 三、功能特点

系统的核心能力围绕以下方面展开。

**智能意图识别**：系统收到用户问题后，通过关键词分析和模型判断自动识别问题类型——属于产品说明书查询、客服政策咨询还是售后工单受理，进而决定后续采用何种处理策略。例如用户问"空气炸锅滤网怎么清洗"，系统判断为说明书查询并触发产品知识检索；用户说"退货"或"退款"，系统则进入售后工单收集流程。

**产品手册检索问答**：系统内置20余类消费电子产品说明书的知识索引，涵盖空气炸锅、洗衣机、微波炉、冰箱、电动牙刷、耳机、相机等常见品类。当用户提出的问题涉及这些产品时，系统从知识库中检索最相关的说明书段落，提取关键信息生成答案，而不是凭空编造。

**多模态图像理解**：用户可以在提问时上传产品图片或故障照片（最多3张），系统通过视觉模型提取图片中的文字、识别产品类型和关键特征，将这些信息融入检索和答案生成。比如用户拍下一张微波炉面板的照片询问某个按钮的功能，系统能识别出微波炉型号并结合对应说明书回答。

**工单收集**：当用户表达报修或退货诉求时，系统不会一次性索要所有信息，而是像人工客服一样逐轮询问——先了解产品型号，再问故障表现，接着确认购买渠道，最后收集联系方式。每轮对话都有明确的提示和确认，用户可以随时取消。

**在线评价查询**：系统支持查询产品的用户评价和口碑信息，帮助消费者了解不同产品的优缺点。例如用户问"XX耳机怎么样"时，系统整合相关评价信息给出客观说明。

**多轮对话记忆**：系统会记住同一会话中的对话历史，不会让用户反复重复已经说过的话。例如用户先说"我买了个空气炸锅"，之后又说"它不工作了"，系统能够理解"它"指的是前面提到的空气炸锅。

**批量处理与离线评测**：系统支持从CSV文件批量导入问题并生成答案，也提供了离线评测工具，方便运维人员对比不同配置下的系统表现，持续优化回答质量。

## 五、系统部署与启动

本章说明如何将系统部署到服务器并启动服务。

部署前请确认已安装Python 3.10及所需依赖包。首先编辑项目根目录下的 `.env` 文件，该文件控制系统的所有运行参数。需要配置的项包括：API服务端口、Bearer Token鉴权密钥、Ollama服务地址、生成模型名称（如qwen2）、嵌入模型名称（如nomic-embed-text或bge-m3）、Milvus数据库连接地址和端口。各项说明已在 `.env` 文件中以注释形式标注。

配置完成后，在终端中进入项目根目录，执行 `uvicorn app.main:app --reload --host 0.0.0.0 --port 8000` 启动API服务。终端会输出uvicorn的启动日志，显示服务监听的地址和端口号。如果配置有误，日志中会提示具体的错误原因。

服务启动后，在浏览器中访问 `http://localhost:8000/healthz`，页面应返回 `{"status":"ok"}`，表示服务运行正常。如果无法访问，请检查端口是否被其他程序占用，或Ollama和Milvus服务是否正常运行。

## 六、API对话接口调用

系统的核心功能通过 `POST /chat` 接口对外提供。调用方需要在HTTP请求头中设置 `Authorization: Bearer <token>`，token值须与 `.env` 文件中配置的 `API_BEARER_TOKEN` 一致。

请求体为JSON格式，包含以下字段：
- `question`（必填）：用户的问题文字，不能为空。
- `images`（可选）：Base64编码的图片列表，最多3张。用户上传产品照片或故障截图时使用。
- `session_id`（可选）：会话标识，用于多轮对话。不传则系统自动生成新会话。
- `stream`（可选）：流式返回开关，当前版本暂不支持，设为 `true` 会返回错误。

以下是一个使用curl调用的示例：

```
curl -X POST http://localhost:8000/chat \
  -H "Authorization: Bearer your_token_here" \
  -H "Content-Type: application/json" \
  -d '{"question": "空气炸锅的滤网怎么清洗？", "session_id": "abc123"}'
```

系统处理成功时，返回JSON格式的响应，`code` 字段为 0，`data.answer` 包含生成的回答文本，`data.session_id` 为当前会话标识。调用方应在后续同一会话的请求中携带相同的 `session_id`，以保持对话上下文连贯。

如果Token无效或缺失，接口返回401状态码。如果 `question` 为空，返回422校验错误。如果 `images` 超过3张，同样返回422错误。调用方应根据错误提示修正请求参数后重试。

## 七、产品手册索引构建

系统的知识库需要在首次部署时通过索引构建脚本初始化，后续新增产品手册时也需要重新构建索引。

首先将产品说明书的 `.txt` 文件放入手册目录（默认 `./手册/`）。系统支持中英文手册，文件编码建议使用UTF-8。手册文件中可以包含 `<PIC>` 标记表示图片位置，系统在解析时会自动处理图片与文本的对应关系。

确认Milvus向量数据库服务已启动并可连接后，在项目根目录的终端中执行 `python scripts/build_index.py`。脚本启动后会依次完成以下步骤：扫描手册目录中的所有 `.txt` 文件、解析文本内容并识别标题和图片标记、按段落边界切分为检索块、调用嵌入模型为每个文本块生成向量、将文本块和向量写入Milvus数据库。

终端会实时输出每个手册文件的解析进度、生成的文本块数量以及写入状态。如果某个文件解析失败，日志中会显示具体的错误原因，但不会中断其他文件的处理。全部完成后，终端输出构建成功摘要，列出总文件数和总块数。

索引构建完成后，可以通过API对话接口发送一个与新增手册相关的问题，验证是否能检索到正确的手册内容。如果检索结果不理想，可以检查手册文本格式是否正确，或调整 `.env` 中的检索参数。

## 八、批量问题处理

当需要批量处理大量用户问题时（例如竞赛提交或效果评估），可以使用批量处理功能。

首先准备一个CSV格式的输入文件，文件至少包含两列：`id`（问题编号）和 `question`（问题文本）。确保Ollama和Milvus服务均正常运行，然后在终端中执行 `python scripts/run_batch_submission.py`。

系统会逐条读取问题，调用对话流水线生成答案。终端显示每道题的处理进度，包括当前题号、路由判断结果、检索耗时和生成耗时。处理过程中如果某条问题出错，系统会记录错误信息并继续处理后续问题。

全部处理完成后，终端输出结果文件的保存路径和总处理条数。输出文件包含 `id` 和 `ret`（答案）两列，可直接用于提交或分析。

## 九、离线评测

系统内置了离线评测工具，用于在标注好的评测集上衡量系统性能，帮助判断配置调整或模型升级的实际效果。

评测集为JSONL格式文件，每行一个JSON对象。每个对象至少需要 `id` 和 `question` 字段，还可以包含 `category`（问题类别）、`should_use_rag`（是否应走检索的金标）、`gold_manual_name`（金标手册名）等标注字段，用于计算路由准确率和检索命中率。

在项目根目录的终端中执行 `python -m eval.run_eval --version <版本标识>` 启动评测。`--version` 参数用于标识本次评测的版本名称，结果文件会以此命名。加上 `--no-judge` 参数可以跳过LLM打分环节，仅统计路由和检索指标，运行速度更快。

评测过程中终端输出每条样本的简要结果。完成后在 `eval/results/` 目录下生成两类文件：`detail_<版本>.csv` 记录每条样本的详细得分、耗时、路由决策和检索信息；`summary_<版本>.csv` 提供汇总统计，包括平均分、检索命中率、路由准确率等。通过对比不同版本的summary文件，可以判断每次改动的效果。

## 十、MCP网关管理

MCP网关是一个独立的Java服务组件，用于统一管理和调度下游工具服务，为客服系统提供地图查询、网页内容抓取、代码沙盒执行等扩展能力。网关需要单独部署。

在 `mcp-gateway` 目录下执行 `docker compose -f docker-compose.java-mcp-gateway.yml up -d --build`，该命令会构建网关镜像并启动Gateway和Nginx两个容器。构建过程约需2-3分钟，首次运行时间较长因为需要下载基础镜像。

容器启动后，执行 `docker compose ps` 确认两个容器的状态均为Running。然后访问 `http://localhost:18080/health` 验证网关健康状态，正常时返回健康检查通过的信息。

网关启动后，可以通过其MCP接口查询已注册的下游服务列表、查看每个服务提供的工具、测试单个工具调用。如果需要在Cursor或其他Agent工具中使用网关能力，在对应配置中挂载 `http://<服务器IP>:18080/mcp` 地址即可。网关目前默认注册了高德地图（地理查询、天气）、网页抓取（公网URL内容提取）和代码沙盒（Docker容器中执行代码）三个下游服务。

如果不需要地图功能，可以跳过API Key配置。停止网关服务时，在 `mcp-gateway` 目录下执行 `docker compose down` 即可。

## 十一、常见问题解答

**API接口返回401 Unauthorized错误怎么办？**

检查请求头中的Authorization字段格式是否正确，须为 `Bearer <token>`，且token值与 `.env` 文件中 `API_BEARER_TOKEN` 配置一致。如果token确实正确，确认 `.env` 文件修改后已重启API服务。

**系统回答与产品完全无关怎么办？**

首先确认对应产品的说明书文件已通过索引构建脚本导入Milvus数据库。可以发送一条明确包含产品名称和功能关键词的问题做对比测试。如果仍未命中，可能需要在 `.env` 中调整检索参数（如降低 `RETRIEVER_CONTEXT_FILTER_SCORE_THRESHOLD` 阈值），或运行离线评测检查检索命中率的统计分布。

**Ollama服务连接失败怎么办？**

执行 `ollama serve` 确认Ollama服务已启动并监听在11434端口。检查 `.env` 中 `OLLAMA_BASE_URL` 是否正确（本地默认为 `http://127.0.0.1:11434`）。确认所需模型已下载：执行 `ollama list` 查看已安装模型，缺少时通过 `ollama pull <模型名>` 下载。

**启动服务后healthz接口无响应？**

检查终端日志中的错误信息。常见原因：Python依赖包未完整安装（执行 `pip list` 检查fastapi、uvicorn、pymilvus等包是否存在）、`.env` 文件格式有误（变量名拼写错误或值格式不对）、端口8000被占用（修改 `.env` 中 `APP_PORT` 为其他端口后重启）。

**如何新增一种产品的说明书？**

将产品说明书的 `.txt` 文件放入 `./手册/` 目录，文件编码使用UTF-8。然后运行 `python scripts/build_index.py` 重建索引。脚本会扫描目录中所有文件，新增的手册内容会被解析、切块并写入Milvus。索引重建完成后，新手册内容即可被检索到，无需重启API服务。

## 十二、术语表

| 术语 | 解释 |
|------|------|
| RAG（检索增强生成） | 先从知识库中检索与问题相关的文档片段，再将其作为上下文提供给语言模型生成答案，提高回答的准确性和可追溯性。 |
| 向量数据库（Milvus） | 存储和检索向量数据的专用数据库。本系统中用于存储产品手册文本块的语义向量，支持相似度搜索。 |
| 意图路由 | 系统自动判断用户问题属于何种类型（说明书查询、客服咨询、售后工单等），并据此选择不同的处理策略。 |
| BM25 | 一种基于关键词匹配的稀疏向量检索算法，适合精确关键词匹配场景，与Dense语义检索互补。 |
| MCP（Model Context Protocol） | AI模型与外部工具之间通信的标准化协议。本系统通过MCP网关统一管理高德地图、网页抓取等工具。 |
| 工单收集 | 当用户表达售后诉求时，系统通过多轮对话逐步收集产品型号、故障表现、购买渠道、联系方式等信息形成工单。 |
| 会话ID（Session ID） | 标识同一用户连续对话的唯一标识符。系统通过会话ID维护对话历史，实现上下文连贯的多轮对话。 |
